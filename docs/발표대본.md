# 발표 대본 — AI 락스크린: 넛지 시스템 & LLM 위젯 배치

> 예상 발표 시간: 8~10분 (데모 포함 시 12~15분)
> 구성: 도입 → 아이디어 1 (넛지) → 아이디어 2 (LLM 위젯 배치) → 기술 스택 → 차별점 → 마무리

---

## 0. 도입 (약 1분)

안녕하세요. 저희가 발표할 프로젝트는 **"AI 락스크린"** 입니다.

여러분, 스마트폰을 하루에 몇 번 켜시나요? 평균적으로 사람들은 하루 80번 이상 화면을 켭니다. 그리고 그때마다 가장 먼저 보는 화면이 바로 **잠금화면**입니다. 하지만 지금의 잠금화면은 어떤가요? 시계 하나, 알림 몇 줄. 하루에 80번을 보는 화면치고는 너무 수동적입니다.

저희는 이 잠금화면을 **"사용자를 행동으로 이끄는 능동적인 화면"** 으로 바꾸는 두 가지 아이디어를 제안합니다.

1. **넛지(Nudge) 시스템** — 쏟아지는 알림 속에서 "행동이 필요한 메시지"만 AI가 골라내고, 바로 실행할 수 있는 버튼을 잠금화면에 띄워줍니다.
2. **LLM 위젯 배치** — "운동할 때 필요한 거 배치해줘"라고 말 한마디만 하면, AI가 알아서 위젯과 바로가기를 골라 화면에 배치해줍니다.

이 두 가지를 차례로 보여드리겠습니다.

---

## 1. 아이디어 ① — 잠금화면 넛지 시스템 (약 3분)

### 1-1. 문제 정의

단체 채팅방을 떠올려 보세요. 하루에 수백 개의 메시지가 오지만, 그중 정말 중요한 건 몇 개 안 됩니다.

> "ㅋㅋㅋㅋ", "오 대박" … 그리고 그 사이에 묻혀 있는 **"토요일 저녁 6시에 강남역 한우집 어때?"**

이 메시지를 놓치면 약속을 놓칩니다. 봤더라도, 일정에 등록하려면 잠금을 풀고 → 캘린더 앱을 열고 → 날짜를 찾고 → 직접 입력해야 합니다. 저희는 이 과정을 잠금화면에서 **버튼 하나**로 줄였습니다.

### 1-2. 동작 시연 (데모 멘트)

*(데모 화면을 보여주며)*

지금 친구가 카톡으로 "토요일 저녁 6시 강남역 한우집 어때?"라고 보냈습니다. 잠금화면을 보시면 —

- 알림은 갤럭시 One UI처럼 **앱 → 채팅방 → 메시지** 3단 구조로 정리되어 있습니다.
- 잡담 메시지들은 그냥 일반 메시지로 표시되지만, 약속이 담긴 이 메시지에는 AI가 자동으로 **[일정 추가] [지도 열기]** 버튼을 달아놨습니다.
- **[일정 추가]** 를 누르면, AI가 메시지에서 추출한 시각(토요일 18시)이 자동으로 채워진 캘린더 등록 화면이 바로 뜹니다.
- **[지도 열기]** 를 누르면, "강남역 한우집"이라는 정제된 장소명으로 지도 검색이 바로 실행됩니다. 발신자 이름이나 조사("어때?") 같은 군더더기는 AI가 알아서 걸러냅니다.

채팅방 안에서는 **[전체 / 행동만]** 필터로, 행동이 필요한 메시지만 모아볼 수도 있습니다.

### 1-3. 어떻게 동작하는가 (기술 설명)

파이프라인은 이렇습니다.

```
시스템 알림 수신 (NotificationListenerService)
  → 알림 저장소 (StateFlow 기반 Repository)
  → AI 분석 (NudgeAnalyzer)
      ① 온디바이스 Gemini Nano 우선
      ② 불가 시 클라우드 Gemini 2.5 Flash Lite로 폴백
  → 넛지 변환 → 잠금화면 UI 표시 → 시스템 액션 실행
```

여기서 강조하고 싶은 설계 포인트가 세 가지 있습니다.

**첫째, 정규식이 아니라 전부 AI 판단입니다.** "토요일", "6시" 같은 패턴 매칭이 아니라, LLM이 문맥을 보고 "이건 실제 행동으로 이어질 메시지인가?"를 판단합니다. 그래서 "다음 주에 한번 보자~" 같은 빈말과 진짜 약속을 구분할 수 있습니다. AI는 important 여부, 액션 종류, 정제된 장소명, ISO 형식의 시작 시각까지 **구조화된 JSON**으로 돌려줍니다.

**둘째, 온디바이스 우선 하이브리드 구조입니다.** 알림은 개인정보 그 자체이기 때문에, 기기 안의 **Gemini Nano**(ML Kit GenAI Prompt API)로 먼저 분석합니다. Nano를 지원하지 않는 기기에서는 클라우드 API로 자동 폴백합니다. 화면 상단의 엔진 칩으로 지금 어떤 엔진이 도는지 — **초록(온디바이스 Nano) / 파랑(클라우드) / 회색(AI 꺼짐)** — 투명하게 보여줍니다.

**셋째, 비용을 고려한 이중 필터입니다.** 모든 메시지를 LLM에 던지면 낭비이기 때문에, 길이 같은 저비용 필터로 먼저 거르고 후보만 AI에 보냅니다. 분석이 실패하면 넛지를 안 띄우면 그만이라, 조용히 실패하는(fail-silent) 구조로 사용자 경험을 해치지 않습니다.

## 2. 아이디어 ② — LLM 기반 위젯 자동 배치 (약 3분)

### 2-1. 문제 정의

두 번째 아이디어입니다. 위젯은 유용하지만, 설정이 귀찮습니다. 어떤 위젯이 있는지 목록을 뒤지고, 하나씩 끌어다 놓고, 크기를 조절하고… 대부분의 사용자는 그냥 포기합니다.

저희의 접근은 간단합니다. **"원하는 상황을 말로 하면, AI가 배치까지 해준다."**

### 2-2. 동작 시연 (데모 멘트)

*(데모 화면을 보여주며)*

배치 시트를 열고 이렇게 입력해 보겠습니다: **"운동할 때 필요한 거 배치해줘"**

그러면 AI가 두 단계로 일합니다.

- **1단계 — 무엇이 필요한가:** 기기에 설치된 앱·위젯·시스템 바로가기 전체 카탈로그를 보고, "운동"이라는 맥락에 맞는 후보 — 시계, 음악, 손전등, 소리 모드 — 를 골라냅니다.
- **2단계 — 어디에 놓을 것인가:** 고른 것들을 화면의 각 영역에 배치합니다. 위젯 트레이에는 알람과 음악 위젯, 좌측 바로가기에는 소리 모드, 우측에는 손전등, 이런 식으로요.

그런데 바로 배치해버리면 사용자가 당황하겠죠. 그래서 저희는 **"고스트 미리보기"** 를 만들었습니다. 추천된 위젯들이 반투명한 윤곽선 형태로 화면에 미리 떠 있고, 격자 탐색 기반 충돌 회피 알고리즘이 기존 위젯과 겹치지 않는 자리를 자동으로 찾아줍니다. 마음에 들면 **수락** — 그 순간 실제 위젯으로 바인딩됩니다. 마음에 안 들면 거절하고 다시 요청하면 됩니다. **AI가 제안하고, 결정은 사용자가 한다** — 이것도 일종의 넛지 철학입니다.

### 2-3. 한 걸음 더 — AI 스케치 위젯

여기서 한 단계 더 나간 기능이 있습니다. 기존 위젯 중에 마음에 드는 게 없다면? **AI가 위젯을 그려줍니다.**

사용자가 원하는 정보 항목(예: 온도, 습도)과 분위기를 말하면,

1. LLM이 **장면을 설계**합니다 — 이미지 프롬프트와 함께, 텍스트가 들어갈 빈 패드의 위치(상대좌표)까지 정해줍니다.
2. Gemini 이미지 모델이 투명 배경의 위젯 이미지를 생성합니다.
3. 그 위에 실시간 데이터 텍스트를 오버레이합니다.

이때 이미지 안에 "온도", "WEATHER" 같은 글자가 구워져 버리면 안 되기 때문에, 프롬프트에 **"텍스트 패드는 반드시 비워둘 것"** 이라는 안전 접미사를 강제로 붙여서 텍스트 영역을 보호합니다. API 키가 없는 환경에서는 규칙 기반 슬롯 레이아웃으로 폴백해서, AI 없이도 항목 수와 화면 비율에 따라 적절한 배치가 나오도록 설계했습니다.

## 3. 기술 스택 & 아키텍처 요약 (약 1분)

- **플랫폼:** Android (Kotlin + Jetpack Compose), 단방향 데이터 흐름 (StateFlow → UI)
- **AI:**
  - 온디바이스 — Gemini Nano (ML Kit GenAI Prompt API)
  - 클라우드 — Gemini 2.5 Flash Lite (구조화 JSON 출력), Gemini 2.5 Flash Image (위젯 이미지 생성)
- **시스템 연동:** NotificationListenerService(알림 수집), AppWidgetHost(실제 앱 위젯 호스팅), CalendarContract / geo URI(넛지 액션), UsageStats(사용 통계 — 추후 추천 컨텍스트로 활용 예정)
- **데모/검증 장치:** 실제 API 없이도 전체 흐름을 보여줄 수 있도록 ▲ 직접 메시지를 발행해 넛지를 트리거하는 테스트 입력 ▲ Nano 분석 테스트용 더미 채팅방 4종 ▲ 위젯 배치 더미 케이스 17종을 내장했습니다.

## 4. 차별점 정리 (약 1분)

| | 기존 잠금화면 | 저희 프로젝트 |
|---|---|---|
| 알림 | 시간순 나열, 읽고 끝 | AI가 행동 필요 메시지만 선별 + 원탭 액션 |
| 위젯 설정 | 사용자가 직접 탐색·배치 | 자연어 한 줄 → AI 추천 → 고스트 미리보기 → 수락 |
| 프라이버시 | — | 온디바이스 Nano 우선, 클라우드는 폴백 |
| AI 투명성 | — | 엔진 상태 칩으로 어떤 AI가 도는지 항상 표시 |
| 제어권 | — | AI는 제안만, 최종 결정은 항상 사용자 |

핵심 메시지는 하나입니다. **"하루 80번 보는 화면을, 보기만 하는 화면에서 행동하는 화면으로."**

## 5. 향후 계획 & 마무리 (약 30초)

앞으로는,

- 사용 통계와 시간대·위치 컨텍스트를 LLM 입력에 반영해 **상황 인지형 자동 배치**(아침엔 출근 모드, 밤엔 수면 모드)로 확장하고,
- 넛지 액션을 일정·지도 외에 송금, 답장, 예약 등으로 넓히고,
- 사용자의 수락/거절 피드백을 학습 신호로 활용해 추천 정확도를 높일 계획입니다.

잠금화면은 스마트폰에서 가장 자주 보이지만 가장 활용되지 않는 화면입니다. 저희는 그 화면에 AI를 더해, 알림을 **행동으로**, 설정을 **대화로** 바꿨습니다.

들어주셔서 감사합니다. 질문 받겠습니다.

## 부록 A. 예상 질의응답 (Q&A 대비)

**Q. 알림을 AI가 읽는 건 프라이버시 문제가 없나요?**
A. 그래서 온디바이스 Gemini Nano를 1순위로 설계했습니다. Nano 지원 기기에서는 메시지가 기기 밖으로 나가지 않습니다. 클라우드는 폴백일 뿐이고, 현재 어떤 엔진이 동작 중인지 화면의 칩으로 항상 표시해 사용자가 인지할 수 있게 했습니다.

**Q. 모든 메시지마다 LLM을 호출하면 비용/배터리가 부담되지 않나요?**
A. 저비용 사전 필터(메시지 길이 등)로 후보를 먼저 거르고, 통과한 메시지만 분석합니다. 온디바이스 Nano가 동작하면 API 비용은 0이고 지연도 1초 미만입니다.

**Q. AI가 잘못 판단하면요? (오탐/미탐)**
A. 넛지는 어디까지나 '제안'입니다. 잘못 떠도 버튼을 안 누르면 그만이고, 분석 실패 시에는 넛지를 표시하지 않는 fail-silent 구조라 일반 알림 경험은 그대로 유지됩니다. 위젯 배치도 고스트 미리보기에서 거절하면 아무 변화가 없습니다.

**Q. 왜 정규식/규칙 기반이 아니라 LLM인가요?**
A. "다음에 밥 한번 먹자"와 "토요일 6시 강남역 어때?"는 패턴이 비슷하지만 의미가 다릅니다. 규칙으로는 이 문맥 구분이 사실상 불가능하고, LLM은 장소명 정제·시각 추론(ISO 8601 변환)까지 한 번에 처리합니다.

**Q. 실제 위젯을 잠금화면에 어떻게 띄우나요?**
A. Android의 AppWidgetHost API로 다른 앱의 실제 위젯을 호스팅하고, showWhenLocked 액티비티로 잠금화면 위에 표시합니다. 권한이 필요한 바인딩은 시스템 다이얼로그로 사용자 동의를 받습니다.

## 부록 B. 데모 진행 순서 (시연용 체크리스트)

1. **넛지 데모** — 테스트 입력창에 "토요일 저녁 6시 강남역 한우집 어때?" 입력 → 넛지 버튼 생성 확인 → 엔진 칩(온디바이스/클라우드) 언급 → [일정 추가] 탭 → 캘린더에 시각 자동 입력된 것 보여주기 → 뒤로 → [지도 열기] 탭 → 정제된 장소명 검색 확인
2. **잡담 대비** — "ㅋㅋㅋ 오늘 날씨 좋다" 같은 잡담 입력 → 넛지가 생기지 않음을 보여주며 "AI가 문맥을 구분한다" 강조
3. **위젯 배치 데모** — 배치 시트 열기 → "운동할 때 필요한 거 배치해줘" 입력 (네트워크 불안 시 더미 케이스 17종 중 '운동' 케이스 사용) → 고스트 미리보기 등장 → 수락 → 실제 배치 확인
4. **(여유 시) AI 스케치** — 정보 항목 입력 → 생성된 위젯 이미지 + 텍스트 오버레이 보여주기

> 팁: 시연 기기 네트워크가 불안하면 모든 단계를 더미/테스트 모드로 진행할 수 있도록 설계되어 있습니다. 발표 전 리허설에서 Nano 모델 다운로드 상태(첫 실행 시 백그라운드 다운로드)를 미리 확인하세요.
